P0013 [Heading 1] 1. Resumen Ejecutivo
P0014 [] HUASCARAN es una aplicación de escritorio para Windows que integra un sistema de Punto de Venta (POS) y gestión de inventario, diseñada para pequeñas y medianas empresas del mercado peruano. Permite administrar productos, ventas, compras, clientes, caja, créditos y facturación electrónica desde un único entorno, con soporte para acceso multi-dispositivo en red local.
P0017 [Heading 1] 2. Arquitectura del Sistema
P0018 [Heading 2] 2.1 Visión General
P0019 [] El sistema adopta una arquitectura de tres capas ejecutadas dentro de un único proceso Electron en la máquina del cliente:
P0021 [List Paragraph] Capa de presentación: React 18 + Vite 5 + Tailwind CSS, servida como archivos estáticos compilados.
P0022 [List Paragraph] Capa de aplicación: API REST construida con Express 5, corriendo en el puerto 3020.
P0023 [List Paragraph] Capa de datos: Base de datos SQLite gestionada con better-sqlite3.
P0025 [] Electron actúa como contenedor del proceso Node.js (backend) y abre una ventana BrowserWindow apuntando a los archivos estáticos del cliente. En modo no-Electron (navegador directo), el sistema puede ser accedido desde otros dispositivos en la red local mediante la IP del servidor.
P0027 [Heading 2] 2.2 Stack Tecnológico
P0029 [Heading 2] 2.3 Diagrama de Capas
P0031 [Heading 2] 2.4 Comunicación Frontend ↔ Backend
P0032 [] Durante el desarrollo, Vite hace proxy de todas las llamadas /api/* al servidor Express en el puerto 3020, eliminando problemas de CORS. En producción (Electron), el frontend cargado como archivos estáticos se comunica directamente con el servidor embebido.
P0034 [] Puerto de desarrollo frontend: 5200 (Vite)
P0035 [] Puerto del servidor API: 3020 (Express)
P0036 [] Proxy en desarrollo: /api/* → http://localhost:3020
P0037 [] CORS: Habilitado para acceso desde red local (otros dispositivos)

P0045 [Heading 1] 4. API REST — Referencia de Endpoints
P0046 [Heading 2] 4.1 Autenticación
P0047 [] El sistema utiliza JSON Web Tokens (JWT) con firma HMAC-SHA256. El token se incluye en cada solicitud protegida como header HTTP:

P0088 [Heading 1] 6. Seguridad
P0089 [Heading 2] 6.1 Autenticación y Autorización
P0090 [List Paragraph] Las contraseñas se almacenan con hash bcrypt (salt automático). Nunca se guardan en texto plano.
P0091 [List Paragraph] La sesión se maneja mediante JWT firmados con HMAC-SHA256. El secreto es configurable por variable de entorno (JWT_SECRET).
P0092 [List Paragraph] Los tokens expiran en 12 horas. No existe mecanismo de refresh token; el usuario debe volver a autenticarse.
P0093 [List Paragraph] Todas las rutas /api/* (excepto las explícitamente públicas) están protegidas por el middleware requireAuth.
P0095 [Heading 2] 6.2 Roles de Usuario
P0097 [Heading 2] 6.3 Protección del Código
P0098 [List Paragraph] El código del servidor se compila con bytenode a formato .jsc (bytecode de V8), impidiendo la lectura directa del código fuente en el entorno del cliente.
P0099 [List Paragraph] El módulo de licencia (licencia.jsc) contiene la lógica de activación y no es legible ni modificable.
P0101 [Heading 2] 6.4 Content Security Policy
P0102 [] El servidor Express define una política CSP en cada respuesta HTTP, permitiendo únicamente:
P0103 [List Paragraph] Scripts y estilos del propio dominio.
P0104 [List Paragraph] Fuentes de Google Fonts (fonts.googleapis.com y fonts.gstatic.com).
P0105 [List Paragraph] Conexiones externas exclusivamente hacia api.nubefact.com (facturación electrónica).
P0108 [Heading 1] 7. Sistema de Licencias
P0109 [] HUASCARAN implementa un control de activación por máquina. Cada instalación está ligada al hardware específico del equipo mediante un identificador de máquina (machine ID) único.
P0111 [Heading 2] 7.1 Flujo de Activación
P0112 [List Paragraph] El cliente instala el sistema. Al primer inicio, el sistema detecta que no existe licencia válida.
P0113 [List Paragraph] El sistema muestra el código de máquina al cliente (visible también en /api/licencia/estado).
P0114 [List Paragraph] El proveedor genera una clave de activación vinculada a ese código de máquina.
P0115 [List Paragraph] El cliente ingresa la clave en la pantalla de activación (POST /api/licencia/activar).
P0116 [List Paragraph] El sistema valida la clave y escribe el archivo de licencia local.
P0117 [List Paragraph] El sistema queda desbloqueado hasta la fecha de expiración definida en la clave.
P0119 [Heading 2] 7.2 Comportamiento con Licencia Vencida o Ausente
P0120 [List Paragraph] Todas las rutas /api/* (excepto auth, licencia y heartbeat) devuelven HTTP 403 con el mensaje de bloqueo y el machine ID.
P0121 [List Paragraph] La ruta /api/facturacion está exenta para preservar el acceso a comprobantes ya emitidos.
P0122 [List Paragraph] El frontend detecta el estado bloqueado y muestra la pantalla de activación al usuario.
P0125 [Heading 1] 8. Sistema de Backups
P0126 [] El servidor implementa tres niveles de copia de seguridad automática de la base de datos SQLite, usando la API nativa de backup de better-sqlite3 (que garantiza consistencia transaccional):
P0129 [] El nombre de cada archivo incluye timestamp completo: db_backup_YYYY-MM-DD_HH-MM-SS[_sufijo].db, permitiendo identificar el momento exacto de cada respaldo.
P0132 [Heading 1] 9. Facturación Electrónica
P0133 [] El módulo de facturación está integrado con Nubefact, un OSE (Operador de Servicios Electrónicos) autorizado por SUNAT para la emisión de comprobantes electrónicos en el Perú.
P0135 [Heading 2] 9.1 Comprobantes soportados
P0136 [List Paragraph] Boleta de Venta Electrónica
P0137 [List Paragraph] Factura Electrónica (requiere RUC del cliente)
P0138 [List Paragraph] Nota de Crédito (anulación o ajuste de comprobantes)
P0140 [Heading 2] 9.2 Integración técnica
P0141 [List Paragraph] La comunicación con Nubefact se realiza mediante su API REST (JSON V1).
P0142 [List Paragraph] Las credenciales (token Nubefact, RUC emisor, series) se configuran en la tabla config.
P0143 [List Paragraph] La ruta /api/facturacion es pública respecto al middleware de licencia para garantizar disponibilidad.
P0144 [List Paragraph] El frontend genera el PDF del comprobante usando html2canvas para su impresión o envío.

P0213 [Heading 1] 13. Control de Errores y Logging
P0214 [Heading 2] 13.1 Logging de errores
P0215 [List Paragraph] El servidor captura todas las excepciones no controladas (uncaughtException) y las escribe en app-error.log con timestamp ISO 8601.
P0216 [List Paragraph] console.error está interceptado para que todos los errores también queden en el log de archivo.
P0217 [List Paragraph] El archivo de log se ubica junto a la base de datos (configurable con DB_PATH).
P0219 [Heading 2] 13.2 Middleware de errores
P0220 [] Express cuenta con un middleware centralizado de manejo de errores (errorHandler) registrado al final de la cadena de middlewares, que captura cualquier error no tratado en los routers y devuelve una respuesta JSON estructurada al cliente.
P0222 [Heading 2] 13.3 asyncHandler
P0223 [] Los controladores de rutas utilizan un wrapper asyncHandler que captura errores de funciones async y los pasa al middleware de errores, evitando try-catch repetitivos en cada endpoint.
